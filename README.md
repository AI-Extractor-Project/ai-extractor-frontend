# AI-Powered Website Audit Tool (Frontend)

This is the frontend implementation for the AI-Native Software Engineer 24-Hour Assignment. It is a lightweight, responsive web application built to analyze a single URL, display factual metrics, and present AI-generated insights and recommendations.

## Tech Stack

- **Framework:** [Next.js 14+ (App Router)](https://nextjs.org/)
- **Language:** TypeScript
- **Styling:** Tailwind CSS
- **State Management:** Zustand (`useAuditStore`)
- **API/Backend Integration:** Express (Backend) & Google Gemini 2.5 Flash

## Getting Started

### Prerequisites
- Node.js (v18 or higher)
- npm or yarn

### Installation & Setup

1. Clone the repository and navigate to the frontend directory:
   ```bash
   cd ai-audit-frontend
   ```

2. Install dependencies:
   ```bash
   npm install
   ```

3. Configure Environment Variables:
   Create a `.env` or `.env.local` file in the root directory and add your backend API URL if needed (by default it will call your backend).

4. Run the development server:
   ```bash
   npm run dev
   ```

5. Open [http://localhost:3000](http://localhost:3000) in your browser.

## Architecture & Engineering Clarity

- **Clean separation between scraping and AI analysis:** The `ResultsPage` enforces a hard visual and component-level boundary. Factual scraped data is exclusively rendered in `<MetricsPanel />`, while AI inferences and recommendations are exclusively rendered in `<InsightsPanel />`. There is no blending of raw facts and AI opinions.
  
- **Structured Outputs:** The frontend is strictly typed using TypeScript interfaces (`PageMetrics`, `AuditInsights`) to consume the structured JSON outputs generated by the AI backend. The UI naturally maps these structured fields into isolated, highly readable components.
  
- **State Management:** Handled by a global Zustand store (`useAuditStore`), which manages the lifecycle of the audit (loading, success, error) and stores the fetched metrics and insights without overcomplicating the prop tree.

## AI Design Decisions & Grounding

- **Visual Grounding:** Every AI insight displayed in the UI explicitly references the exact metric it is grounded on (e.g., `Metric = 0 CTAs`). This creates trust and adheres to the requirement that AI must be grounded in structured factual data.
  
- **Actionable AI Outputs:** Recommendations are sorted by priority and visually distinguished, making the AI's structured output immediately actionable and concise for the user.

## Thoughtful Trade offs (Frontend)

- **Client-Side State vs Server-Side Routing:** For the scope of this assignment, the audit results are kept in client-side state (Zustand) rather than relying on Server-Side Rendering (SSR) with URL parameters (e.g. `?url=...`). This trade-off kept the solution simple and practical, allowing for rapid execution within the 24-hour limit, though it means results are lost on a hard page refresh.
  
- **Custom Design vs Component Library:** A custom Tailwind CSS design system was built from scratch instead of relying on a heavy component library (like MUI or Bootstrap). This trade-off prioritized a highly specific, premium, "AI-native" aesthetic over out-of-the-box components, demonstrating stronger raw frontend engineering skills.
  
- **Single API Call vs Streaming:** The frontend awaits a single resolved JSON payload from the backend instead of utilizing Server-Sent Events (SSE) or UI streaming. This is a practical trade-off that significantly reduces architectural complexity while still delivering a fast result.

## What I would improve with more time

- **Persistent Shareable Results:** Use URL routing parameters (e.g., `?auditId=123`) to fetch and render reports so they can be shared via link and persist across reloads.
- **Caching & History:** Implement local storage or a database integration to save audit history and avoid re-analyzing the same URL multiple times.
- **Loading Skeletons:** Add deeper loading skeleton states while the AI is processing the response to improve perceived performance during the wait time.
- **Enhanced Error Handling:** Provide more granular visual feedback if the backend scraping fails (e.g., site blocks bots).
